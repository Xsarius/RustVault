# ADR-0003: API Design

- **Status:** Accepted
- **Date:** 2026-03-03
- **Deciders:** RustVault core team
- **Tags:** api, rest, openapi, websocket

## Context

RustVault's API is the contract between the Rust backend and all clients (web SPA, iOS, Android). It must:

- Support CRUD operations for all domain entities (transactions, accounts, budgets, rules, etc.)
- Handle bulk operations efficiently (import pipeline, batch categorization)
- Provide real-time feedback (import progress, live updates)
- Be self-documenting and explorable
- Prevent data leakage between users in multi-user setups

## Decisions

### Protocol: REST + WebSocket

- **REST** for all CRUD operations — stateless, cacheable, well-understood
- **WebSocket** for real-time features — import progress streaming, live dashboard updates

No GraphQL — the domain model is well-defined and doesn't benefit from client-driven query shapes. REST keeps the backend simpler and avoids N+1 query complexity.

### Specification: OpenAPI 3.1 (auto-generated)

API schemas are generated from Rust types using **utoipa**. The spec is served at `/api/docs` via **Scalar** (interactive UI). This ensures documentation is always in sync with code.

### URL Structure

```
/api/auth/...           # Authentication (login, register, OIDC)
/api/banks/...          # Bank CRUD
/api/accounts/...       # Account CRUD
/api/transactions/...   # Transaction CRUD + bulk ops
/api/transfers/...      # Transfer CRUD
/api/imports/...        # Import pipeline (upload, map, confirm)
/api/categories/...     # Category CRUD
/api/tags/...           # Tag CRUD
/api/rules/...          # Auto-categorization rules
/api/budgets/...        # Budget CRUD + forecasting
/api/reports/...        # Aggregation endpoints
/api/export/...         # Data export (CSV, JSON, QIF)
/api/settings/...       # User + instance settings
/api/ws                 # WebSocket for real-time events
```

### Versioning: None (always backwards-compatible)

RustVault is a self-hosted application with a single backend serving its own frontend. There is no public API contract with third-party consumers. The API is always backwards-compatible — breaking changes are avoided by adding new fields/endpoints rather than modifying existing ones. No `/v1/` prefix to keep URLs clean and routing simple.

### Pagination: Cursor-based

Cursor-based pagination avoids the consistency issues of offset-based pagination when data changes between pages. All list endpoints return `{ data: T[], next_cursor: string | null }`.

### Filtering & Sorting

Query parameters on list endpoints: `?date_from=&date_to=&category_id=&search=&sort=date:desc`. Complex filters use structured query params rather than request bodies on GET.

### Authentication

Every request (except `/auth/login`, `/auth/register`, `/auth/oidc/*`) requires a valid JWT in the `Authorization: Bearer <token>` header. See ADR-0006 for error handling on auth failures.

### Response Envelope

Consistent response shape across all endpoints:

```json
// Success
{ "data": { ... }, "meta": { "cursor": "...", "total": 42 } }

// Error
{ "error": { "code": "VALIDATION_FAILED", "message": "...", "details": [...] } }
```

## Consequences

### Positive

- Self-documenting API — always-current OpenAPI spec at `/api/docs`
- Type safety end-to-end — Rust types → OpenAPI → TypeScript client (generated)
- Cursor pagination handles large datasets without skipping/duplicating rows
- WebSocket enables real-time import UX without polling
- Consistent response envelope simplifies client-side error handling

### Negative

- REST can be verbose for complex queries (mitigated: structured filter params)
- OpenAPI generation adds build-time dependency on `utoipa` annotations
- WebSocket adds connection management complexity

### Risks

- REST can be verbose for complex queries (mitigated: structured filter params)
- OpenAPI generation adds build-time dependency on `utoipa` annotations
